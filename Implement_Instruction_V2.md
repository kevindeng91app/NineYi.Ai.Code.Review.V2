# NineYi.Ai.CodeReview.V2 實作說明書

---

## 一、站台核心價值

本站台旨在將「Pull Request / Merge Request 的人工 Code Review」轉化為**可配置、可追蹤成本、可稽核、可跨平台**的 AI Review 流程。

| 特性 | 說明 |
|------|------|
| **跨平台一致化** | 同一套審查流程同時支援 GitHub / GitLab / Bitbucket，不因平台而碎裂 |
| **以 Repo 為單位的治理** | 不同 Repository 可套用不同審查規則（Rules），讓規則可管理、可調整 |
| **可控成本** | 記錄 Dify API 使用量與費用，避免 AI 審查成為不可控支出 |
| **提升 Prompt/規則品質** | 統計哪些 rule 最常觸發、哪類檔案常出問題，用數據回饋優化 prompt |
| **安全前置** | 敏感關鍵字檢測（密碼、金鑰…）作為最基本的安全護欄 |
| **稽核與除錯** | 完整 ReviewLog/檔案級紀錄，出事可追溯，快速定位是哪一步、哪條規則、花了多少錢 |

## 二、完整規格


### 1. Webhook 接收與路由

- **單一 Controller**：`WebhookController`
- **三個 Action**：
  - `POST /api/webhook/github`
  - `POST /api/webhook/gitlab`
  - `POST /api/webhook/bitbucket`
- **責任**：僅負責將各平台 payload 解析為一致的 Application 輸入模型，不承擔後續業務流程

---

### 2. Application 輸入模型

**統一入口**：`StartCodeReviewCommand`（非 Domain Entity）

**最小必要欄位**：

| 欄位 | 說明 |
|------|------|
| `ProviderType` | 平台識別 |
| `RepoFullName` | Repo 識別 |
| `PullRequestNumber` | PR/MR 識別 |
| `Title` | PR 標題，用於忽略 auto PR |
| `PullRequestRef` | PR 參考資訊 |
| `DiffUrl` | 優先使用，直接帶入模型 |
| `HeadCommitSha` | Source/Head commit hash，用於抓 raw content（後續需要） |

> **確立原則**：DiffUrl 帶進模型，client 取得 diff 時優先用 DiffUrl。

---

### 3. RepoHost Client 抽象與 Factory

```
IRepoHostClientFactory
└── Get(ProviderType) → IRepoHostClient

IRepoHostClient
├── GitHubClient
├── GitLabClient
└── BitbucketClient
```

- **`IRepoHostClientFactory`**：依 `ProviderType` 回傳對應的 `IRepoHostClient`
- **`IRepoHostClient`**：統一以 `PullRequestRef` 作為輸入，平台差異封裝於各 client 內

---

### 4. Diff 取得、重試策略、回傳格式

#### 4.1 Diff 取得策略

1. 優先使用 `PullRequestRef.DiffUrl`
2. 失敗時考慮 fallback（未定案），但明確需要 retry 機制

#### 4.2 Retry 需求

| 錯誤類型 | 處理方式 |
|----------|----------|
| timeout、網路錯誤、5xx、429 | **暫時性錯誤**，retry 3 次 |
| 401、403、404 | **非暫時性錯誤**，不做無意義 retry |

#### 4.3 Retry 放置方式

- **共用 HttpClient policy**：處理共通暫時性錯誤
- **各平台 client 特化**：處理平台特性（如 rate limit header）

#### 4.4 回傳格式（逐檔集合）

`IRepoHostClient` 回傳「**檔案變更集合**」而非單一 diff 字串，每個 item 包含：

| 欄位 | 說明 |
|------|------|
| `FileName` | 檔案名稱 |
| `FilePath` | 檔案路徑 |
| `FileType` | 副檔名，用於篩選 rules（如 `.cs`） |
| `Diff` | 該檔案的 diff 內容 |
| `ChangedLineRangesNew` | 從 diff hunks 解析的 new file 端 + 行號範圍 |

---

### 5. Client 回傳 diff 集合前的檔案過濾

在各平台 client 組裝 diff 集合時，先過濾掉不需要審查的內容：

- 排除 AutoGeneratedContent
- 排除特定路徑（如 `/AutoGenerated/`、`/i18n/`）
- 排除特定副檔名/檔案類型（視需求）

---

### 6. CodeReviewService 主流程

`CodeReviewService.StartAsync(command)` 步驟：

```
Step 1: PR Title 忽略規則（省成本 gate）
         ├── 由 config 提供 ignore keywords
         ├── 匹配方式：Contains 命中即忽略
         └── 命中則 skip，不抓 diff、不打 Dify

Step 2: Resolve 對應平台 client
         └── client = factory.Get(command.ProviderType)

Step 3: 取得逐檔 diff 集合
         ├── 優先用 DiffUrl
         └── 暫時性錯誤 retry 3 次

Step 4: 取得每個檔案的 raw content
         ├── 以 PR 分支的 head/source commit 抓取
         └── 送 Dify input = diff + raw content + 檔案資訊 + 行號範圍

Step 5: Repo 白名單驗證
         ├── 依 repo enum 判斷 RepoName 是否在白名單內
         └── 通過才進行「撈 rules → 打 Dify」

Step 6: 從 SQL Server 依 RepoName 撈 rules
         ├── DifyApiKey
         └── 可處理的檔案類型（如 .cs）

Step 7: 以檔案為主，逐條 rule 呼叫 Dify
         ├── 依 FileType 篩選 rules
         ├── 對每條 rule 呼叫一次 Dify API
         └── 把回應整理成該檔案的 findings 集合

Step 8: 輸出結構化報告（多檔案 × 多 findings）
```

**每筆 Finding 結構**：

| 欄位 | 說明 |
|------|------|
| `RuleId` / `RuleName` | 執行的 rule |
| `Severity` | 嚴重程度 |
| `StartLine` ~ `EndLine` | 行號範圍（以 new file 端 + 行號） |
| `Suggestion` | 改善建議 |

---

### 7. 設定檔（config）需求

| 設定類型 | 說明 | 匹配方式 |
|----------|------|----------|
| PR title ignore keywords | PR 標題忽略關鍵字 | Contains 命中即忽略 |
| 檔案排除規則 | 路徑/資料夾/副檔名，用於 client 回傳前過濾 | 路徑前綴/副檔名比對 |

## 三、實作落地建議

### A. 物件與介面清單

#### API 層

```
WebhookController
├── Github(payload)    → StartCodeReviewCommand → CodeReviewService.StartAsync(command)
├── Gitlab(payload)    → StartCodeReviewCommand → CodeReviewService.StartAsync(command)
└── Bitbucket(payload) → StartCodeReviewCommand → CodeReviewService.StartAsync(command)
```

#### Application 層

```
StartCodeReviewCommand                  # 統一輸入模型
CodeReviewService                       # 核心審查流程編排
ReviewResult                            # 整體審查輸出
FileReviewReport                        # 單檔審查輸出
Finding                                 # 單筆審查發現

Options/
├── PullRequestIgnoreOptions            # Title ignore keywords
└── FileExcludeOptions                  # 排除路徑/副檔名等
```

#### Infrastructure 層

```
IRepoHostClientFactory
IRepoHostClient
├── GitHubClient
├── GitLabClient
└── BitbucketClient
    ├── GetPullRequestDiffFilesAsync(PullRequestRef pr)
    │   └── → IEnumerable<FileDiffItem>
    └── GetFileRawContentAsync(PullRequestRef pr, string path, string commitSha)
        └── → string (raw content)

HttpPolicy
├── 共用 retry policy（暫時性錯誤）
└── 平台特化處理（429 rate limit 等）
```

#### Domain / Persistence 層

```
Entities/
├── Repository, Rule, RepositoryRuleMapping
├── ReviewLog, ReviewFileLog
└── HotKeyword, RuleStatistics, DifyUsageLog

Repositories/
└── SQL Server：repo → rules 查詢 repository
```

---

### B. Diff → 行號範圍解析

解析 unified diff 每個 hunk header：

```
@@ -oldStart,oldCount +newStart,newCount @@
```

**行號範圍取 new 端**：

- `start = newStart`
- `end = newStart + newCount - 1`

> **多 hunk 處理**：同一檔案多 hunk 建議存成多段 range（更精準），但送 Dify 時仍以整檔 diff + raw content 為單位。

---

### C. Dify 呼叫與結構化回傳

**每次 call Dify 帶入**：

| 欄位 | 說明 |
|------|------|
| rule 資訊 | 用於 prompt/上下文 |
| file path / name / type | 檔案識別資訊 |
| diff | 整檔 diff |
| raw content | 整檔原始內容 |
| changed line ranges | 解析的 +new 行號範圍 |

**回傳後整理為 findings**：

```json
{
  "ruleId": "...",
  "ruleName": "...",
  "severity": "Error | Warning | Info",
  "startLine": 10,
  "endLine": 15,
  "suggestion": "..."
}
```

---

### D. 記錄與統計

| 時機 | 寫入資料表 | 內容 |
|------|-----------|------|
| 每次 PR 審查 | `ReviewLog` | 含 skipped 原因 |
| 每個檔案/每條 rule | `ReviewFileLog` | 結果、嚴重度、行號、建議 |
| 每次 Dify call | `DifyUsageLog` | token 用量、cost |
| 每次 rule 觸發 | `RuleStatistics` | 觸發次數累計 |

---

## 四、端到端流程總覽

```
Webhook（GitHub / GitLab / Bitbucket payload）
    ↓
統一 StartCodeReviewCommand
    ↓
PR Title ignore gate（命中則 skip + 寫 ReviewLog）
    ↓
Resolve IRepoHostClient（依 ProviderType）
    ↓
取得逐檔 diff（DiffUrl 優先，retry 3 次）+ 檔案過濾
    ↓
取得每檔 raw content（head/source commit）
    ↓
Repo 白名單驗證
    ↓
SQL Server 撈 rules（含 DifyApiKey + 可處理檔案類型）
    ↓
逐檔 × 逐 rule 呼叫 Dify API
    ↓
產出結構化 findings（rule / severity / line range / suggestion）
    ↓
寫入 ReviewLog / ReviewFileLog / DifyUsageLog / RuleStatistics
```
